# D2.2 Social Acceptability Workshops: WP5/SUMA Input Brief v0.3

Source: `docs/antifragicity/deliverables/AntifragiCity - D2.2 - Social Acceptability Workshops [DRAFT].pdf`  
Status: draft WP2 deliverable with deliberative social-acceptability evidence.  
Primary WP5 role: methodology, evidence caveats, acceptability/equity constraints, communication requirements, and validation context.

## 1. What This Source Is For

D2.2 should shape how SUMA frames and communicates recommendations. Its value for WP5 is not an automated social score. Its value is a set of constraints and caveats: social acceptability status, equity risk, vulnerable groups, public rationale, reversibility, mitigation, and communication channel requirements.

D2.2 should therefore be treated primarily as methodology and validation context. Only the parts needed by selected response actions or UI/reporting flows should become schemas.

## 4. Methodology Versus Software Objects

| D2.2 content | WP5 classification | Implementation decision |
|---|---|---|
| Citizen forum findings | evidence / validation context | Store as provenance if used; do not hard-code as universal rule. |
| General acceptability principles | methodology | Document as recommendation-status rule and UI warning. |
| Vulnerable group concerns | data contract when pilot-specific | Add fields only when action comparison uses group impacts. |
| Public communication requirements | API/reporting contract | Needed when SUMA exports alerts or recommendation explanations. |
| Social approval of an action | governance | Must remain owner-approved, not automatically inferred by SUMA. |

## 6. T5.2 Development Implications

| SUMA module | What to build | Notes |
|---|---|---|
| Response-action status engine | Combine technical priority with acceptability/equity/mitigation statuses. | Rule-based warnings only, not automated social approval. |
| Explanation payload builder | Generate public and technical explanations from scenario/action metadata. | DMO/UI can consume later. |
| Stakeholder evidence registry | Store evidence origin, sample, representativity, and validation caveats. | Can remain internal in first prototype. |
| Equity warning module | Flag burden shifts and vulnerable-group caveats when data exists. | Requires pilot zone/group definitions. |
| Channel-readiness checker | Check whether app-only communication is insufficient for critical alerts. | Based on pilot communication channels. |

## 8. KPI And Antifragility Link

| Need | D2.2 contribution | Status |
|---|---|---|
| Social legitimacy of adaptation | Acceptability and trust evidence | indicative, context-specific |
| Equity dimension of antifragility | Vulnerable-group and burden-shift constraints | schema now, data-dependent |
| Learning loop | Post-event feedback and updated priorities | records now, automated learning deferred |
| Communication reach | Real-time and non-digital inclusion | implementable as output metadata |
| Recommendation readiness | Mitigation/public rationale/reversibility | implementable as gating rule |

Antifragility claims should not be positive if the system improves aggregate traffic performance by shifting unacceptable burden to vulnerable groups or unserved neighbourhoods. This is a methodological rule, not a standalone numeric equation from D2.2.

## 9. Mini-GA Questions

Ask LISER/WP2, city partners, and triage leads:

- For each proposed response action, is social evidence `assessed`, `indicative_only`, `substituted`, or `missing`?
- Which vulnerable groups or neighbourhoods may be harmed, delayed, excluded, or digitally disconnected?
- Is the action acceptable only with mitigation? If yes, what mitigation and who owns it?
- What public rationale would be needed before deployment?
- Which actions are high technical priority but socially contested?
- What non-digital channels exist for critical alerts in each pilot?
- How should Odesa acceptability be represented given the suspended forum?

Expected Mini-GA output: one row per response action with `acceptability_status`, `evidence_origin`, `equity_risk`, `mitigation_required`, `communication_owner`, and `deployment_readiness`.

## 10. Technology Stack Note

Use Pydantic schemas and FastAPI response models for acceptability/equity metadata, but do not create a separate social-scoring service in D5.1. Store evidence as JSON records with provenance and caveats. In the frontend, render these as warnings, badges, and explanation panels rather than hidden algorithmic scores.
