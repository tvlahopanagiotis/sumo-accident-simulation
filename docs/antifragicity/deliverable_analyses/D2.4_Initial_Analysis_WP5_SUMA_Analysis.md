# D2.4 Initial Analysis: WP5/SUMA Input Brief v0.3

Source: `docs/antifragicity/deliverables/AntifragiCity - D2.4 [INITIAL ANALYSIS].docx`  
Status: initial analysis of mobility/social acceptability survey evidence.  
Primary WP5 role: provisional quantitative evidence for priority objectives, equity gates, explanation/reporting, communication channels, trusted data sources, and evidence caveats.

## 1. What This Source Is For

D2.4 provides a quantitative acceptability envelope for SUMA design. It is useful for shaping defaults and warnings: emergency access should be treated as a high-priority objective, unfair neighbourhood burden should trigger mitigation, explanations and after-action reports should be expected, and app-only communication should not be assumed sufficient.

D2.4 should not be used to hard-code universal thresholds. Percentages must remain evidence records with `sample_n`, `evidence_scope`, `threshold_status`, and `validation_status`.

| Trusted data sources include PT systems, traffic sensors, emergency/road reports, and weather. | Data-use question | Prioritise institutional/operational feeds where available. |
| Mobile phone location and social media are less accepted. | Data-use question | Add privacy/acceptability flags for sensitive sources. |

## 4. Methodology Versus Software Objects

| D2.4 content | WP5 classification | Implementation decision |
|---|---|---|
| Emergency access priority | methodology plus configurable default | Implement `PriorityObjective`; allow pilot override. |
| Survey percentages | evidence | Store as evidence metadata, not code constants. |
| Fairness/mitigation preference | methodology/gating rule | Implement equity warning/gate when spatial burden data exists. |
| Explanation/reporting preference | API/reporting contract | Include explanation/reporting payloads in D5.1 outputs. |

## 9. Mini-GA Questions

Ask LISER/WP2, WP6/pilots, DMO, and AUTH:

- Which emergency corridors, hospital access routes, PT lines, or critical assets should be protected in each pilot?
- Which neighbourhood/zone definitions should be used for equity burden checks?
- What mitigation rule applies if total system performance improves but one area worsens?
- Which communication channels can the pilot actually use: app, SMS, road signs, PT announcements, web, hotline, TV/radio, in-person?
- Which data sources are available, trusted, socially acceptable, and legally usable?
- Who owns public explanations, after-action reports, and feedback moderation?
- Which D2.4 percentages should remain only as evidence and which can become pilot-validated thresholds?

Expected Mini-GA output: one table per pilot with `protected_routes`, `zone_definitions`, `priority_objectives`, `mitigation_rule`, `communication_channels`, `data_source_contracts`, and `explanation/report_owner`.

## 10. Technology Stack Note

Implement D2.4-derived rules as configuration and metadata in the FastAPI/Pydantic layer. Store survey-derived evidence as JSON/YAML fixtures or SQLite records with sample and validation metadata. Use the React UI to show explanation, reporting, and non-digital-channel warnings. Do not encode survey percentages as constants in SUMO simulation logic.
