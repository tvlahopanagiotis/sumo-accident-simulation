# D2.7 KPI Framework: WP5/SUMA Input Brief v0.3

Source: `docs/antifragicity/deliverables/AntifragiCity - D2.7 - AntifragiCity KPIs and Framework Development.pdf`  
Status: WP2 KPI framework and catalogue deliverable.  
Primary WP5 role: KPI registry, KPI selection workflow, KPI observation schema, threshold/baseline governance, KPI calculation metadata, and scenario export semantics.

## 1. What This Source Is For

D2.7 is the primary KPI-contract source for SUMA. It should become a machine-readable KPI registry plus a pilot/use-case KPI selection workflow. It should also define how SUMA stores observations, labels calculation modes, tracks data quality, handles thresholds/baselines, and exports scenario comparison results.

D2.7 is not a finished SUMA API, not a validated demo-site KPI dataset, not a universal mandatory KPI list, and not proof that all 209 catalogue KPIs are implementable in RP1.

## 5. Selection And Data-Quality Calculations

These are SUMA-owned operational formulas for D5.1/T5.2 selection support. They are not claimed as D2.7-validated science.

```text
selection_score =
  wr*relevance +
  wf*feasibility +
  wi*integrability +
  wa*actionability +
  waf*af_required

coverage_ratio = available_required_kpis / required_kpis

data_quality = 0.4*completeness + 0.4*accuracy + 0.2*timeliness
```

Each weight must be configurable and labelled `default`, `pilot_calibrated`, or `assumed`.

## 6. KPI Tiers

| Tier | Meaning | WP5 use |
|---|---|---|
| `top_22_practitioner` | Feasible/practitioner-selected indicators | Useful MVP seed, incomplete for AF. |
| `common_64` | Broader common indicator set if available | Good for cross-pilot comparability. |
| `full_209_catalogue` | Complete D2.7 catalogue | Reference only; not implementation target. |
| `af_required_override` | Network/system/AF indicators required even if difficult | Protects D2.3/SUMA scientific integrity. |
| `pilot_selected` | Confirmed Mini-GA subset per use case | D5.1/T5.2 implementation target. |
| `deferred` | Relevant but not implementable in RP1 | Backlog with owner and rationale. |

Network/system performance KPIs must not be silently dropped because they were omitted from the practitioner top 22. They are essential for resilience/AF interpretation.

## 9. Mini-GA Questions

Ask AUTH/WP2, CU, pilots, WP3, WP4, and Rhoe:

- What is the minimal KPI subset for each retained use case?
- Which KPIs are raw inputs, computed outputs, or model/calibration parameters?
- Which network/system KPIs must be included through `af_required_override`?
- Which Layer 3 SUMA KPIs are defined, method-TBC, baseline-TBC, target-TBC, or owner-missing?
- Who defines KPI targets, warning thresholds, critical thresholds, regulatory baselines, pilot baselines, and scenario baselines?
- What proxy use is acceptable per KPI?
- What dataset, denominator, formula, and validation method supports each selected KPI?
- Which KPIs are real-time, periodic, offline, simulated, proxy, or deferred?

Expected Mini-GA output: `KpiEvidenceLedger` rows for each selected KPI with `id`, `unit`, `layer`, `data_role`, `formula`, `baseline_status`, `threshold_status`, `source_dataset`, `confidence`, `verification_method`, `owner`, and `D5.1/T5.2 status`.

## 10. Technology Stack Note

Use Pydantic models for registry/observation schemas, SQLite or JSON fixtures for the D5.1 catalogue, Pandas/NumPy/SciPy for KPI calculations, and OpenAPI examples for selected KPIs. Do not introduce a KPI microservice before the core FastAPI contract is stable. Add background jobs for expensive KPI calculations in T5.2 if needed.
