# D2.6 Requirement Elicitation: WP5/SUMA Input Brief v0.3

Source: `docs/antifragicity/deliverables/AntifragiCity - D2.6 - Requirement elicitation.pdf`  
Status: WP2 requirement elicitation and specification deliverable for SUMA/KER5.  
Primary WP5 role: authoritative requirements baseline, traceability structure, acceptance criteria, verification methods, and prioritisation evidence for D5.1 specification and T5.2 implementation planning.

## 1. What This Source Is For

D2.6 is the main requirements contract source for SUMA. It consolidates literature, ontology work, partner/pilot consultation, stakeholder input, and a Delphi-inspired expert scoring process into a prioritised SUMA requirement set. It should drive the D5.1 requirement matrix, endpoint acceptance criteria, schema tests, Mini-GA requirement triage, and the T5.2 backlog.

D2.6 is not a proof that all requirements are already feasible in every pilot, not a completed architecture, not a validated implementation, and not permission to silently assign unresolved data, governance, or integration work to Rhoe. Its consensus results indicate perceived importance and agreement, not implementation readiness.

Important internal mapping: D2.6 references WP5 outputs D5.1-D5.4. In this internal documentation, D2.6 references to implementation/integration should be translated into the T5.2 implementation workstream unless the consortium separately asks for deliverable-specific wording.

## 5. Consensus Handling Rule

Use D2.6 consensus status as a governance signal, not as an implementation switch.

| Consensus category | D2.6 evidence | WP5/SUMA treatment |
|---|---|---|
| High consensus | 35 requirements, IQR <= 1.0, Table 6 pp.40-41 | Baseline source for D5.1 scope unless contradicted by feasibility, ownership, or ethics constraints. |
| Moderate consensus | 10 requirements, IQR = 2.0, Table 7 pp.41-42 | Keep in schema/backlog where relevant, but require owner, maturity label, and Mini-GA clarification before strong claims. |
| Low consensus | DEL-R30, IQR = 3.0, Sec. 6 p.41 | Do not ignore it: split governance is operationally important but implementation must be staged and context-specific. |

Moderate-consensus items are especially important for SUMA even if not fully implementation-ready: real traffic data, data provenance/trust, antifragility score calculation, long-term learning, emergency prioritisation, stakeholder input, proactive rerouting, active mobility during disruptions, user preference incorporation, and energy/emissions optimisation.

Open clarification: reconcile the reported 47 Delphi statements with the extracted 35 high-consensus, 10 moderate-consensus, and 1 low-consensus counts before freezing the requirement registry.

## 8. Mini-GA Questions

Ask Cardiff/CU, Rhoe, DMO, pilots, WP3, WP4, AUTH, and city partners:

- Do partners accept the D2.6 FR/DR/IR/UR/NFR/GR set as the D5.1 baseline, or should any requirement be reclassified after the Mini-GA?
- For each retained pilot use case, which D2.6 requirements are essential, deferred, or not relevant?
- Who owns the acceptance criterion for each essential requirement?
- Which D2.6 "Should" requirements must become RP1 commitments because they are critical for a pilot?
- What is the minimum role model needed for UR-01, given low consensus around split-governance usability?
- What target latency windows are realistic for FR-06/NFR-02 in planning mode, near-real-time mode, and degraded mode?
- Which simulator integrations are required as executable T5.2 work and which are adapter contracts only?
- What is the minimum viable antifragility metric operationalisation for FR-03 without overclaiming validation?
- What audit events must be logged for GR-02: scenario run, parameter change, data injection, export, recommendation, stakeholder input, or all of these?
- What data retention/minimisation constraints apply per pilot for GR-01?

## 9. Technology Stack Note

D2.6 strengthens the case for a requirements-first technical stack:

- Use FastAPI/OpenAPI and Pydantic schemas to make FR-05/NFR-01 testable.
- Store the `RequirementDefinition`, `AcceptanceCriterion`, `RequirementGate`, and `VerificationEvidence` registries first as versioned JSON fixtures, then in SQLite when evidence records become mutable.
- Use pytest schema/API tests for FR-01, FR-02, FR-05, DR-02, and GR-02.
- Keep near-real-time support behind explicit runtime modes and latency targets; do not add Redis/Celery only because D2.6 says near-real-time.
- Use the existing documentation page as the human-facing NFR-01 evidence surface, with this D2.6 analysis linked from the curated AntifragiCity documentation index.
