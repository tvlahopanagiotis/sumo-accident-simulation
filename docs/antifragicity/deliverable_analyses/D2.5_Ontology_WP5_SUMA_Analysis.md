# D2.5 Ontology: WP5/SUMA Input Brief v0.3

Source: `docs/antifragicity/deliverables/AntifragiCity - D2.5 - Ontology.pdf`  
Status: WP2 ontology deliverable.  
Primary WP5 role: semantic contract source for identifiers, ontology-aligned DTOs, JSON-LD compatibility, named graph/city graph conventions, KPI observation separation, provenance, and semantic-lift staging.

## 1. What This Source Is For

D2.5 is the semantic backbone for SUMA interoperability. D5.1 should use it to define stable object identifiers, ontology classes, JSON/JSON-LD-compatible payloads, named graph conventions, semantic metadata, and links between events, scenarios, observations, response actions, actors, spatial units, and KPIs.

D2.5 does not require SUMA to implement a full knowledge graph, rule engine, SPARQL endpoint, SWRL reasoning, or SHACL validation in the first API version. D5.1 should define ontology-aligned contracts first; semantic lifting and rule validation can be staged for T5.2.

## 5. Required Semantic Metadata

```yaml
OntologyMetadata:
  iri: string | null
  ontology_class: string
  city_code: string | null
  city_graph: string | null
  local_id: string | null
  external_ids: object
  semantic_version: string
  schema_version: string
  source_ref: string | null
  provenance: object
  quality: object | null
  access_level: public | internal | restricted | confidential | unknown
  redaction_status: none | redacted | aggregated | synthetic | unknown
  mapping_status: mapped | partially_mapped | unmapped | not_applicable
```

Recommended IRI pattern to discuss with CU/WP2:

```text
https://antifragicity.eu/id/{city_code}/{ontology_class}/{local_id}
```

D5.1 should not hard-code this as final unless CU confirms it. It should define a `canonical_id` plus optional `iri` and `external_ids`.

## 6. Key Relations To Represent

| Relation | SUMA use |
|---|---|
| `affects` | event/action affects transport element, actor, spatial unit, KPI, or service |
| `respondsTo` | response action responds to disruption event |
| `triggersResponse` | event/scenario triggers response action |
| `updatesState` | simulation/action updates system state |
| `previousState/nextState` | scenario/run state progression |
| `propagatesTo` | cascading disruption pathway |
| `hasCause` | stressor/event cause relation; use only when causality is validated, otherwise store reported or inferred links with confidence/provenance |
| `locatedIn` | event/asset/actor located in spatial unit |
| `dependsOn` | service/KPI/action dependency |
| `governedBy` | action governed by policy/authority constraint |
| `implementedBy` | action implemented by actor/module |
| `hasObservation` | KPI/state/event linked to observation |
| `evaluatedIn` | KPI/response evaluated in scenario/run |
| `validDuring` | temporal validity |

### 7.4 Minimal JSON-LD And Relation Example

Illustrative only, pending CU confirmation of context and IRI conventions:

```json
{
  "@context": "/api/v1/ontology/context.jsonld",
  "@id": "urn:afc:thessaloniki:DisruptionEvent:event_ahepa_flood_access_001",
  "@type": "DisruptionEvent",
  "eventDomain": "environment_weather",
  "affects": [
    "urn:afc:thessaloniki:TransportElement:ahepa_corridor_primary_001"
  ],
  "validationStatus": "unvalidated"
}
```

Example relation set:

| Subject | Relation | Object | Caveat |
|---|---|---|---|
| `event_ahepa_flood_access_001` | `affects` | `ahepa_corridor_primary_001` | illustrative until AOI confirmed |
| `action_ahepa_signal_priority_001` | `respondsTo` | `event_ahepa_flood_access_001` | candidate action |

| `event_ahepa_flood_access_001` | `hasCause` | `storm_forecast_001` | avoid unless source validates causality |

## 11. Technology Stack Note

Use JSON and JSON-LD first. Add `rdflib` and `pySHACL` only when semantic-lift validation becomes a T5.2 task. Do not introduce a triple store for D5.1. If a KG backend becomes necessary later, evaluate Apache Jena Fuseki or GraphDB after CU confirms ontology artefacts, expected query workload, and hosting responsibility.
