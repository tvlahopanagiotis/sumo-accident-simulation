# D2.1 Urban Event Mapping: WP5/SUMA Input Brief v0.3

Source: `docs/antifragicity/deliverables/AntifragiCity - D2.1 - Urban Event Mapping.pdf`  
Status: completed WP2 deliverable.  
Primary WP5 role: event taxonomy, source crosswalk, event/stressor/disruption semantics, and scenario-fixture basis.

## 1. What This Source Is For

D2.1 is one of the strongest direct inputs to D5.1 because SUMA cannot model, compare, or validate a disruption scenario without a consistent event representation. It should be used as the canonical starting point for disruption-event contracts, scenario triggers, severity/scale metadata, source provenance, and event-data readiness.

D2.1 is not a calibrated operational event model and does not validate antifragility. It should not be used to infer final severity thresholds, event frequencies, or quantitative AF scores without pilot confirmation and D2.3/D2.7 methods.

## 2. Evidence To Preserve In D5.1

| Evidence | Reference | D5.1 implication |
|---|---|---|
| Event taxonomy is organised by domain and scale. | D2.1 Sec. 2.1, Table 1, pp. 12-13 | Define controlled vocabularies for `event_domain` and `event_scale`. |
| Severity/urgency is separate from scale. | D2.1 Sec. 2.1, Table 2, pp. 13-14 | Do not infer severity from spatial scale; expose both fields. |
| Source crosswalks cover survey, social media, EM-DAT/Copernicus, and internal categories. | D2.1 Sec. 2.2, Table 3, p. 14 | Add `taxonomy_version`, `crosswalk_version`, and source-specific mapping metadata. |
| Urban Event Catalogue is a seed catalogue, not a complete pilot dataset. | D2.1 Sec. 4.1, Table 36, p. 77; Sec. 7.3, Table 39, pp. 99-104 | Use catalogue entries as API examples and test fixtures only. |
| Survey and social-media evidence have sampling/source limitations. | D2.1 Sec. 3.3.1.1.2, pp. 18-19; Sec. 3.3.2.6.6, Table 35, pp. 73-74 | Add confidence, source limitations, language/platform, and validation status. |
| Pilot validation was planned after D2.1. | D2.1 Sec. 3.1, pp. 15-16; Sec. 5, p. 81 | Mark pilot event profiles as provisional until city/WP6 confirmation. |
| Event, stressor, disruption, and impact are distinct concepts. | D2.1 Sec. 7.2, Table 38, pp. 97-99 | Use separate fields/relations rather than one flat `event_name`. |

## 5. T5.2 Development Implications

| SUMA module | What to build | Notes |
|---|---|---|
| Event taxonomy registry | Versioned JSON/YAML registry for domains, scales, severity labels, and crosswalks. | Start as file-backed fixture; later DB-backed. |
| Event validation service | Pydantic schema validation plus semantic checks for independent scale/severity. | Expose missing fields and unmapped categories. |
| Scenario trigger builder | Convert validated events/stressors into scenario inputs for SUMO or partner adapters. | Preserve provenance and assumptions. |
| Data-gap tracker | Track pilot event-data gaps from Mini-GA outputs. | Internal planning API or markdown/CSV is sufficient initially. |
| Fixture generator | Convert D2.1 catalogue examples into test payloads. | Must be labelled `catalogue_fixture`, not pilot truth. |

## 7. KPI And Antifragility Link

D2.1 contributes the disturbance side of the AF pipeline. It can populate event domain, scale, severity, source confidence, and impact pathway, but it does not compute resilience or antifragility on its own.

| AF/KPI need | D2.1 contribution | D5.1 status |
|---|---|---|
| Disturbance classification | Domain, subcategory, scale | implementable |
| Disturbance severity | Severity level and basis | implementable, threshold owner needed |
| Disturbance effectiveness for D2.3 | Input to `Deff` mapping | schema now, mapping under D2.3/WP2 decision |
| Exposure/frequency | Catalogue/social/survey signals | proxy only |
| Event-to-impact chain | Affected modes/assets/groups | implementable |
| Validated AF score | none | deferred to D2.3/D2.7/WP6 |

## 8. Mini-GA Questions

Ask CU/WP2 and pilot tables:

- What is the canonical D2.1 domain, subcategory, scale, and severity for each retained use case?
- Is the scenario based on observed evidence, historical replay, forecast, synthetic stress test, or manual assumption?
- Who validates event severity and AOI boundaries per pilot?
- Which source will be used for event detection: operator feed, sensor, weather feed, social/citizen report, manual entry, or partner report?
- Which transport assets, modes, vulnerable groups, and spatial units are affected?
- Which missing event fields should be `acquire`, `proxy`, `defer`, or `drop`?
- Is the D2.1 taxonomy frozen enough for D5.1 examples, or must Rhoe version it as provisional?

Expected Mini-GA output: one row per retained use case with `event_domain`, `event_scale`, `severity_basis`, `trigger_type`, `source_type`, `aoi`, `validation_owner`, and `missing_event_data_action`.

## 9. Technology Stack Note

Use the existing FastAPI/Pydantic backend for event validation and OpenAPI schema generation. Store early taxonomy/crosswalk definitions as versioned YAML/JSON fixtures in the repo. Use GeoJSON in API payloads and plan PostGIS for T5.2 if pilot AOIs, critical corridors, or spatial joins become central. JSON-LD fields should be added as metadata, but a full RDF store is not required for D5.1.
