# D2.3 Equilibrium And Antifragility: WP5/SUMA Input Brief v0.3

Source: `docs/antifragicity/deliverables/AntifragiCity - D2.3 - Establishing urban states of equilibrium.pdf`  
Status: theoretical/methodological framework for equilibrium, resilience, antifragility, and SRI.  
Primary WP5 role: calculation semantics, state-vector structure, maturity labels, and antifragility validation scaffolding.

## 1. What This Source Is For

D2.3 gives SUMA the scientific calculation backbone for equilibrium, resilience, and antifragility. It should inform `SystemState`, normalized KPI vectors, disturbance-effectiveness mapping, target-performance calculation, SRI advisory outputs, feasible-set checks, and AF validation records.

D2.3 must not be presented as empirically validated pilot evidence. In D5.1 and T5.2, every D2.3-derived output needs `calculation_mode`, `model_maturity`, `kpi_coverage_tier`, `parameter_status`, and validation caveats.

## 4. Equations And Calculation Contracts

The exact formula choices should be aligned with D2.3 and partner decisions, but D5.1 should already define the calculation contracts and required metadata. Treat any formula example in this brief as illustrative until weights, windows, thresholds, source reliability values, and validation methods are confirmed by the relevant partners.

### 4.1 System State

```text
G_t = (S_t, u_t)
```

Where `S_t` is the network/system state and `u_t` is the control/intervention state. SUMA should not expose this as an abstract formula only; it should persist the practical references:

```yaml
SystemState:
  id: string
  pilot_id: string
  scenario_id: string
  timestamp: datetime
  topology_version: string
  network_ref: string
  demand_ref: string | null
  capacity_vector_ref: string | null
  flow_vector_ref: string | null
  control_state_ref: string | null
  kpi_vector_id: string | null
  model_maturity: theoretical | benchmark | pilot_calibration | operational | proxy
```

### 4.2 KPI Vector

```text
V_t = [M, R, A, C, Deff, S, Q, P, B, Eenergy, EICT, I]
```

Minimal prototype vector:

```text
V_min = [M, R, C, Deff, S, I]
```

Recommended schema:

```yaml
KpiVector:
  id: string
  state_id: string
  values:
    M: NormalizedIndicator | null
    R: NormalizedIndicator | null
    A: NormalizedIndicator | null
    C: NormalizedIndicator | null
    Deff: NormalizedIndicator | null
    S: NormalizedIndicator | null
    Q: NormalizedIndicator | null
    P: NormalizedIndicator | null
    B: NormalizedIndicator | null
    Eenergy: NormalizedIndicator | null
    EICT: NormalizedIndicator | null
    I: NormalizedIndicator | null
  coverage_tier: minimal | partial | full | insufficient
  missing_indicators: string[]
  normalization_version: string
```

### 4.3 Normalized Indicator

```yaml
NormalizedIndicator:
  raw_value: number | null
  normalized_value: number | null
  unit: string | null
  denominator: string | null
  exposure_basis: string | null
  direction: higher_is_better | lower_is_better | target_band
  formula_ref: string
  source_kpi_ids: string[]
  calculation_mode: observed | simulated | derived | proxy | theoretical
  confidence: number | null
  validation_status: string
```

Normalization must respect directionality:

```text
x_hat_higher_is_better = (x - xmin) / (xmax - xmin)
x_hat_lower_is_better = (xmax - x) / (xmax - xmin)
x_hat_target_band = band_score(x, lower_bound, target_min, target_max, upper_bound)
```

### 4.4 Disturbance Effectiveness

`Deff` should be calculated from D2.1 event classification and source/severity mapping. D5.1 should avoid hard-coding an unapproved equation. Specify the contract:

```yaml
DisturbanceEffectiveness:
  event_id: string
  severity_level: integer | null
  event_scale: string
  source_weight: number | null
  recency_weight: number | null
  exposure_weight: number | null
  deff_value: number | null
  mapping_version: string
  parameter_status: default | calibrated | assumed | missing
```

Mini-GA decision needed: who owns the mapping from D2.1 event domain/scale/severity/source to `Deff`.

Illustrative toy example, not pilot-validated:

```yaml
DisturbanceEffectiveness:
  event_id: event_ahepa_flood_access_001
  severity_level: 4
  event_scale: mid
  source_weight: 0.6
  recency_weight: 1.0
  exposure_weight: null
  deff_value: 0.45
  mapping_version: illustrative_default_v0
  parameter_status: assumed
```

### 4.5 Antifragility Validation

Use D2.3 validation as a gated batch workflow:

```text
AF = mu_post / E_base
```

Illustrative AF validation record. This record deliberately remains `not_validated` because the causal and equity checks are missing.

```yaml
AFValidationRecord:
  id: af_ahepa_mock_001
  pilot_id: ahepa
  scenario_id: scenario_ahepa_flood_intervention_001
  baseline_state_id: state_ahepa_baseline_mock
  post_event_window:
    start_day: 31
    end_day: 60
  E_base: 0.62
  mu_post: 0.66
  AF: 1.06
  confidence_interval:
    lower_95: null
    upper_95: null
  stability_passed: null
  causal_passed: false
  equity_passed: null
  validation_decision: not_validated
  caveats:
    - illustrative_only
    - missing_confidence_interval
    - missing_equity_check
    - missing_causal_attribution
```

Where `E_base` is the baseline equilibrium/performance value and `mu_post` is the mean post-event or post-intervention value over the agreed validation window. The existing analysis referred to a 30-day post-event window (`mu30`); D5.1 should make the window configurable and explicitly record it.

```yaml
AFValidationRecord:
  id: string
  pilot_id: string
  scenario_id: string
  baseline_state_id: string
  post_event_window:
    start_day: integer
    end_day: integer
  E_base: number | null
  mu_post: number | null
  AF: number | null
  confidence_interval: object | null
  stability_passed: boolean | null
  causal_passed: boolean | null
  equity_passed: boolean | null
  validation_decision: not_validated | proxy | pilot_calibration | validated | failed
  caveats: string[]
```

## 8. Technology Stack Note

Use Python with NumPy, Pandas, and SciPy for D2.3 calculations. Use NetworkX and SUMO-derived network outputs for redundancy/connectivity prototypes where possible. Run heavy/batch calculations through a background-job layer in T5.2, ideally Celery plus Redis, while D5.1 can specify asynchronous jobs without implementing the full queue. Store calculation metadata through Pydantic models and OpenAPI schemas. Avoid embedding formulas directly in the frontend.
